Product

What is MinNotes?

MinNotes strips a notes app down to one question: what are you thinking about? You type it. The app figures out the rest.

Under the hood, every submission is sent to a Cloudflare Worker, which forwards it to Groq's llama-3.3-70b-versatile model. The model reads the text and returns a decision: is this a plain NOTE to keep, or a REMINDER that should fire at a specific time? If it's a reminder, the app builds a local push notification via expo-notifications and schedules it.

The whole product hinges on one pattern — "AI as UX": the model replaces a category dropdown, a date-picker, and a recurrence selector. The user just writes naturally ("remind me to call mom sunday", "buy milk", "the meeting moved to 3pm") and the app does the right thing.

Three tabs make up the entire interface — Write, Reminders, Calendar — rendered in a committed pure black-and-white aesthetic. Everything is local-first: notes live in AsyncStorage and never leave the device except for that one-shot AI analysis. No account, no cloud, no sync. Yet.

Strengths

The clever bits — what's genuinely good.

✓AI decides note-vs-reminder

A real modern UX move. The user never touches a category picker, a date field, or a recurrence dropdown — the model infers intent from plain language. This is the entire product thesis and it's executed cleanly.

✓Graceful AI fallback

If Groq fails or the worker 500s, the text still saves as a plain note. The try/catch in App.tsx means a broken edge never blocks the user from capturing a thought.

✓Local-first by default

Notifications fire offline, and notes never leave the device except for the one-shot AI analysis. No account wall, no network dependency to read your own data.

✓Lean dependency surface

Minimal deps, strict TypeScript, no Redux/Zustand/recoil bloat. State stays simple because the data model is simple — a list of notes and a list of reminders.

✓Recurring reminders done right

Daily / weekly / monthly recurrence is handled correctly through expo-notifications triggers — not reinvented with a fragile cron of setTimeouts.

✓Draft auto-persists

The current draft is saved while typing, so an accidental swipe or kill never loses the half-finished thought you were mid-sentence on.

✓Committed monochrome aesthetic

The pure black / white / dim-gray look isn't a theme toggle — it's the whole identity, applied consistently across every screen. Restraint reads as confidence.

✓Tight, focused codebase

~1450 lines for a full shipped concept. No framework ceremony, no over-abstraction. What's there does exactly what it needs to.

Critical · Fix today

🚨 Critical security issue

🚨

The Groq API key is hardcoded and committed

Critical

The production Groq API key lives in plaintext inside worker/wrangler.toml, checked into the repository. Anyone with read access to the repo — or anyone who simply discovers the Worker's public URL — can call it and drain your Groq quota or abuse the key for their own inference workloads, billed to you.

worker/wrangler.toml · line leaked
GROQ_API_KEY = "gsk_..."

How to fix — in order

  1. Rotate the key NOW. Revoke this one in the Groq dashboard and generate a fresh one. The leaked key must be considered permanently compromised.
  2. Move it to a secret. Use wrangler secret put GROQ_API_KEY (production) or a gitignored .dev.vars file (local dev). Never commit it again.
  3. Purge it from git history. Even after deletion, it lives in prior commits. Use git filter-repo or BFG to scrub every occurrence, then force-push.
  4. Lock the worker down with an origin allowlist + rate limiting (see the HIGH issue below).

This is the #1 thing to fix. Everything else is secondary.

Audit

Issues & weaknesses

Every finding, tagged by severity. No invented problems — these are the real ones.

Critical

Leaked API key

The Groq key is committed in plaintext in worker/wrangler.toml. See the critical section above — rotate, secret-ify, scrub history.

High

No auth + wide-open CORS

The Worker reflects any incoming Origin header straight back into Access-Control-Allow-Origin and has no rate limiting. Result: it's effectively an open Groq proxy. Anyone can hit your Worker URL and burn your (now hopefully rotated) quota.

High

No backend persistence

All notes and reminders live only in AsyncStorage on the device. Lose the phone, wipe the app, or switch devices and everything is gone. There is no sync, no backup, no export. For a notes app this is an existential data-loss risk.

High

Race conditions in AsyncStorage

The send() path does a classic read-modify-write on NOTES_KEY with no lock. Two submissions landing near-simultaneously will both read the same list, each append, and the second write silently overwrites the first. Data disappears with no error.

Medium

App.tsx is 1243 lines

WritePad, RemindersPad, MonthView, the App shell, and all styles are crammed into a single file. It works, but it's the single biggest maintainability drag in the project. Split into dedicated components.

Medium

Timezone ambiguity

The system prompt says "assume 9am local", but the LLM has no idea what timezone the user is in. The Worker sends now in UTC. So "tomorrow" gets interpreted relative to… what, exactly? This is a real scheduling-bug waiting to fire at the wrong hour.

Medium

No schema validation on LLM output

The Worker JSON.parses Groq's response with no validation. A malformed or hallucinated reply throws and returns a 500. The app falls back gracefully (good), but the Worker should validate the shape before responding — and ideally retry once.

Medium

Unused dependencies

react-native-reanimated is wired into babel.config.js but never imported in App.tsx — that's roughly 4MB of dead weight in the bundle. expo-keep-awake, expo-blur, expo-asset, and expo-font look unused too.

Medium

Empty catch blocks

Silent catch {} blocks appear around six times across the codebase. Swallowed errors hide real bugs — a failed parse or a denied permission looks identical to success. At minimum, log them.

Low

No tests at all

Zero test files. For a side project that's survivable; for something heading to the store, the notification-trigger logic and the AI-parsing path deserve coverage.

Low

No error boundary

A single render-time throw crashes the entire app to a red screen. A top-level error boundary would at least keep the user in a recoverable state.

Low

Hardcoded Worker URL

The Worker endpoint is hardcoded in src/api.ts. It should be env-driven (expo-constants or an .env) so dev/staging/prod can differ without a code change.

Low

Dead "deleting" spinner

The deleting loading state is set and then cleared synchronously in the same tick, so the spinner never actually paints. It's dead UX code — and it appears three times.

Low

Calendar inconsistency

MonthView draws note-dots on calendar days, but tapping a day shows an event list that only includes reminders, never notes. The dots promise something the list doesn't deliver.

Low

Wrong LICENSE author

The LICENSE still reads "Copyright (c) 2015-present 650 Industries, Inc. (aka Expo)" — copy-pasted straight from the Expo template. It should bear the actual author's name.

Low

No "mark as done"

Reminders can only be deleted — there's no way to mark one complete. For a reminders app that's a notable UX gap.

Low

Fired reminders linger forever

One-shot reminders that have already fired stay in the list indefinitely. There's no expiry or auto-archive, so the list only grows.

Low

Duplicate permission prompt

Notification permissions are requested once on app startup and again when creating the first reminder. One well-timed prompt is better UX.

The brain

The AI prompt, annotated

The system prompt that lives in worker/src/index.ts. Each line paired with what it does — and where it's fragile.

worker/src/index.ts · SYSTEM PROMPTllama-3.3-70b
1You are an assistant that classifies a user's thought.
2Decide if it is a NOTE or a REMINDER.
3A NOTE is information with no time component.
4A REMINDER references a time, date, or event.
5For reminders, extract title + ISO datetime.
6Assume 9am local if no time is specified.
7Detect recurrence: daily / weekly / monthly.
8Be conservative — when unsure, prefer NOTE.
9Return ONLY valid JSON, no prose.
10{ "type", "title", "datetime", "recurrence" }
11Current time (UTC): {now}
RoleSets the model's single job. Clean and scoped — no multi-turn confusion.
Decision ruleThe core classifier. Two buckets, clearly defined. Good.
⚠ Timezone bug"9am local" — but local to whom? The model doesn't know the user's timezone. Combined with UTC now on line 11, "tomorrow" is ambiguous. Send the user's IANA timezone.
RecurrenceMaps cleanly to expo-notifications triggers. Solid.
Tuning knob"Be conservative" is the bias dial. Crank it stricter if too many notes become accidental reminders.
⚠ FragileNo schema enforcement here. The Worker trusts the model to return valid JSON. Validate after parse — and retry once on failure.
⚠ UTC contextOnly signal the model gets about "now". Insufficient for correct local-time scheduling across timezones.

Bottom line

Final verdict

B−

Clever product. Real security debt.

MinNotes is the rare side project where the core idea is actually better than the execution needs to be — "an AI that decides note-vs-reminder" is a sharp, defensible product thesis, and the local-first, monochrome build commits to it fully. The codebase is lean, typed, and refreshingly free of framework bloat. What holds it back from a higher grade is one critical, unmissable security hole (a committed API key), the absence of any persistence beyond the device, and a 1243-line god file that will only get harder to maintain. None of that is hard to fix. The hard part — designing something people would actually want to use — is already done.

Not store-ready until security is fixed · concept is strong
